<objective>
Add a "Day wrap time" setting (HH:MM, default "00:00") that treats any completed entry whose local start time falls before that time as belonging to the previous day — i.e., excludes it from the current note's import.

Purpose: Night-owl users who work past midnight want entries at e.g. 01:00–03:59 to count as the previous day, not the new one. Default "00:00" is a no-op so existing behaviour is unchanged.
Output: dayWrapTime in settings interface + filter in fetchTimeEntries + settings UI field + tests.
</objective>

In `fetchTimeEntries`, after the running-entry filter (`const completed = raw.filter(...)`) and before the sort, add the day wrap filter:

```typescript
// Day wrap time: exclude entries whose local start is before the configured boundary
const [wrapH, wrapM] = plugin.settings.dayWrapTime.split(':').map(Number);
const wrapMinutes = (wrapH ?? 0) * 60 + (wrapM ?? 0);
const wrapped = wrapMinutes === 0
  ? completed
  : completed.filter(e => {
      const startLocal = new Date(e.start);
      return startLocal.getHours() * 60 + startLocal.getMinutes() >= wrapMinutes;
    });
```

Then use `wrapped` instead of `completed` in the sort and map steps. Keep the variable rename minimal — just replace `completed` with `wrapped` from this point forward (or sort/map `wrapped` in place).

Key constraints:
- When `dayWrapTime` is `'00:00'`, `wrapMinutes` is 0 and the fast path (`wrapMinutes === 0`) skips filtering entirely — no behaviour change for default users.
- Use `new Date(e.start)` to get local time; `getHours()` and `getMinutes()` return local time in the user's timezone.
- Do not change the date range query — we still fetch the full day. The filter is client-side.

Add a new `Setting` for "Day wrap time" between the Sort order setting and the Columns heading. Use a text input with placeholder `00:00`:

```typescript
new Setting(containerEl)
  .setName('Day wrap time')
  .setDesc('Entries starting before this time (HH:MM) are treated as the previous day. Default 00:00 disables this.')
  .addText(text => text
    .setPlaceholder('00:00')
    .setValue(this.plugin.settings.dayWrapTime)
    .onChange(async (value) => {
      this.plugin.settings.dayWrapTime = value;
      await this.plugin.saveSettings();
    })
  );
```

No validation needed — invalid values will produce NaN from parseInt, which the filter handles gracefully (0 * 60 + NaN = NaN, NaN >= 0 is false, so entries would be excluded; a future improvement could add validation but is out of scope here). Actually, let's add minimal safety: in api.ts the `?? 0` fallback already handles NaN from `Number('') === 0` — so empty string defaults to 00:00 safely.
