<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Obsidian Sync → cloud | Files under `.obsidian/plugins/<id>/data.json` are included in Obsidian Sync. Anything written there travels to Obsidian's cloud. |
| data.json → disk | Plaintext JSON on device disk; readable by any process with filesystem access. |
| localStorage → disk | Obsidian stores localStorage in a per-device SQLite-backed store under the vault's `.obsidian` folder; NOT included in Obsidian Sync. Still plaintext at rest, but scope is device-local. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Information Disclosure | API token stored in `data.json` (traverses Obsidian Sync) | mitigate | Remove `apiToken` from `TogglImportSettings` (Task 1); all reads/writes go through `Plugin.getApiToken()` / `app.saveLocalStorage('toggl-api-token', ...)`. `loadSettings()` actively strips any legacy `apiToken` key from prior `data.json` and rewrites the sanitized object (RESEARCH.md Pitfall 5). |
| T-08-02 | Information Disclosure | Legacy `data.json` on upgrading users still contains plaintext token | mitigate | `loadSettings()` sanitization runs on first load after upgrade, deletes the legacy field, and calls `saveSettings()` to persist the cleaned JSON. Test in `tests/main.test.ts` asserts the saved payload has no `apiToken`. |
| T-08-03 | Information Disclosure | Token leaked into error messages or logs | mitigate | Existing `tests/api.test.ts` "does not include API token in error messages" test remains in force after Task 2 refactor — the mock still provides `test-token-abc123` and asserts it is not in error output. No new log/throw sites added in Plan 01. |
| T-08-04 | Tampering | Untrusted process with filesystem access modifies the localStorage entry | accept | Single-user desktop plugin; user's device is the trust boundary (per ASVS V4 NO in RESEARCH.md). Encrypting localStorage is out of scope and not supported by the Obsidian API. |
| T-08-05 | Repudiation | None | accept | No audit/logging requirement for single-user plugin (per ASVS V3 NO in RESEARCH.md). |
| T-08-06 | Denial of Service | Corrupted localStorage entry returns unexpected shape from `loadLocalStorage` | mitigate | `getApiToken()` casts via `as string | null` and falls back to `''`. Empty-token guard in `editorCallback` catches the empty case and shows a user-facing notice instead of crashing. |
| T-08-07 | Elevation of Privilege | N/A — no privilege model | accept | Single-user plugin; no privilege escalation surface. |
| T-08-08 | Spoofing | N/A — no identity system beyond the Toggl API token itself | accept | Token IS the identity; authentication is handled by Toggl API server response (401 path already covered in `api.ts`). |

**Severity summary:** All high-severity Information Disclosure threats (T-08-01, T-08-02, T-08-03) are `mitigate` with specific tasks. No high-severity threats remain with `accept` disposition. Plan 01 is the implementation backbone for SEC-02 — without it, T-08-01 stays open.
