# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Scaffolding. Generated by gsd-ui-researcher.
>
> **Phase 1 note:** This phase produces zero user-visible UI. It creates the plugin skeleton (src/main.ts, manifest.json, esbuild config, tsconfig). The design contract here establishes the inherited design system and foundational tokens that all subsequent phases MUST follow. Executor: do not add any UI in this phase.

## Spacing Scale

Declared values (must be multiples of 4). All spacing in plugin UI uses Obsidian's CSS custom properties — do not hardcode pixel values in plugin stylesheets.

| Token | Value | Obsidian CSS Var | Usage |
|-------|-------|-----------------|-------|
| xs | 4px | `var(--size-4-1)` | Icon gaps, inline padding |
| sm | 8px | `var(--size-4-2)` | Compact element spacing |
| md | 16px | `var(--size-4-4)` | Default element spacing, setting row padding |
| lg | 24px | `var(--size-4-6)` | Section padding |
| xl | 32px | `var(--size-4-8)` | Layout gaps |
| 2xl | 48px | `var(--size-4-12)` | Major section breaks |
| 3xl | 64px | `var(--size-4-16)` | Page-level spacing |

Exceptions: none for Phase 1 (no visible UI exists in this phase)

**Source:** Default — Obsidian CSS variable naming convention; 8-point scale applied

## Typography

All typography inherits from the active Obsidian theme. Plugin code must not override `font-family`, `font-size`, or `color` on container elements — use Obsidian's semantic CSS classes and variables only.

| Role | Size | Weight | Line Height | Obsidian CSS Var |
|------|------|--------|-------------|-----------------|
| Body | 14px | 400 | 1.5 | `var(--font-text-size)` |
| Label | 13px | 400 | 1.4 | `var(--font-smaller)` |
| Heading | 16px | 600 | 1.2 | `var(--font-ui-medium)` |
| Display | 20px | 600 | 1.2 | `var(--h4-size)` |

**Source:** Default — Obsidian standard typography scale; plugin settings tabs use 14px body universally

---

## Color

All colors are inherited from the active Obsidian theme via CSS custom properties. The plugin MUST NOT hardcode hex values in styles.css — use Obsidian semantic color variables exclusively.

| Role | Value | Obsidian CSS Var | Usage |
|------|-------|-----------------|-------|
| Dominant (60%) | theme-defined | `var(--background-primary)` | Main note pane, settings container background |
| Secondary (30%) | theme-defined | `var(--background-secondary)` | Settings tab sidebar, modal header |
| Accent (10%) | theme-defined | `var(--interactive-accent)` | Toggle switches (on state), active setting highlight |
| Destructive | theme-defined | `var(--text-error)` | API token warning text (SET-02), error notices |

Accent reserved for: toggle switch active state only. Do not apply accent color to text labels, borders, or decorative elements.

**Source:** Default — Obsidian CSS variable system; 60/30/10 split maps to Obsidian's background-primary/secondary/accent hierarchy

## Copywriting Contract

Phase 1 has no user-visible text. The table below pre-defines copy for the plugin's silent load state and establishes conventions for all phases.

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — Phase 1 has no CTA (commands registered in Phase 5) |
| Plugin load notice | None — plugin loads silently; no Notice on `onload` in Phase 1 |
| Empty state heading | Not applicable in Phase 1 |
| Empty state body | Not applicable in Phase 1 |
| Error state | Not applicable in Phase 1 (no API calls, no user input) |
| Destructive confirmation | Not applicable in Phase 1 (no destructive actions) |

**Convention established for all phases:**
- All Notice text uses sentence case, not title case
- Error notices identify the problem and state the next step: e.g. "Invalid date format. Open a daily note named yyyy-mm-dd."
- No exclamation marks in error copy

**Source:** Default — standard Obsidian plugin copy conventions

## Phase-Specific Constraints

These constraints apply to Phase 1 execution and must not be violated:

1. **No UI code in Phase 1.** `src/main.ts` must not call `this.addSettingTab()`, `this.addCommand()`, `new Notice()`, or any DOM manipulation. These are Phase 2 and Phase 5 concerns.
2. **No styles.css content in Phase 1.** The file may be created as an empty file (required by Phase 6 release assets) but must contain no rules.
3. **Plugin loads silently.** `onload()` calls only `this.loadSettings()`. No user-visible output.
4. **DEFAULT_SETTINGS must be typed to the full TogglImportSettings interface.** Even though settings UI is Phase 2, the interface and defaults are established here so later phases can rely on a stable contract.
