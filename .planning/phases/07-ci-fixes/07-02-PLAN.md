<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub push/PR → GitHub Actions runner | Untrusted code from PRs runs in CI context |
| GitHub Actions runner → npm registry | Install step fetches packages over the network |
| GitHub Actions runner → GitHub Releases API | Release step writes to repo releases (release.yml only) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-05 | Elevation of Privilege | ci.yml job permissions | mitigate | Omit the `permissions:` block in ci.yml entirely. GitHub defaults for untrusted PR workflows are read-only, which is correct for build+test. Explicit write permissions only in release.yml where they're needed. |
| T-07-06 | Tampering | Third-party action versions | mitigate | Pin all actions to major version tags (`@v4` for checkout/setup-node, `@v2` for action-gh-release) matching release.yml convention. No floating `@main` or `@master` refs. |
| T-07-07 | Denial of Service | CI queue by malicious PRs | accept | Public repo default concurrency is acceptable for a single-maintainer project. No concurrency group needed at v1.1 scale. |
| T-07-08 | Information Disclosure | Logs leaking secrets | mitigate | ci.yml has no secrets reads and no env block. release.yml only reads the default `GITHUB_TOKEN` used by softprops/action-gh-release — already established and unchanged. Node version bump does not affect secret handling. |
| T-07-09 | Tampering | PR from fork can modify ci.yml and run arbitrary code | accept | Standard GitHub Actions behavior: `pull_request` (not `pull_request_target`) runs in a read-only, secret-less context for forks. We use `pull_request`, which is the safe default. |
| T-07-10 | Repudiation | Release tag pushed without passing CI | mitigate (future) | Out of scope for this phase. Current flow: CI runs on push to main, release runs on tag. A future phase could add `workflow_run` gating if desired. Documented as accepted for v1.1. |
</threat_model>
